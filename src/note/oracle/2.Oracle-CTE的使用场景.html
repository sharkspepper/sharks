什么是 CTE？

CTE（Common Table Expression）是一个临时的、命名的结果集，它存在于单个SQL语句的执行范围内（SELECT / INSERT / UPDATE / DELETE）。
你可以把它看作“一个可以在主查询中多次引用的临时视图”，极大提升复杂查询的可读性和可维护性。

WITH cte_name AS (
    SELECT col1, col2 FROM table WHERE condition
)
SELECT * FROM cte_name;
✅ 核心优势：步骤化、可复用、支持递归、逻辑清晰
🔧

1️⃣ 复杂查询步骤拆分（告别嵌套地狱）

当SQL嵌套超过3层，可读性断崖式下跌。CTE让每一步都有名字，像写文档一样写SQL。

-- ❌ 传统嵌套写法（难以维护）
SELECT department, AVG(salary)
FROM (
    SELECT department, salary,
           ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as rn
    FROM (
        SELECT * FROM employees WHERE status = 'active'
    ) active_emp
) ranked
WHERE rn <= 3
GROUP BY department;

-- ✅ CTE 分步优雅版
WITH active_employees AS (
    SELECT * FROM employees WHERE status = 'active'
),
ranked_employees AS (
    SELECT department, salary,
           ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as rn
    FROM active_employees
)
SELECT department, AVG(salary)
FROM ranked_employees
WHERE rn <= 3
GROUP BY department;
🌳

2️⃣ 递归CTE · 树形/层级数据

组织架构、菜单树、评论链、地铁线路、物料清单（BOM）—— 递归CTE是唯一内置解法。

WITH RECURSIVE dept_tree AS (
    -- 锚点：顶级部门
    SELECT dept_id, dept_name, parent_id, 1 as level
    FROM departments
    WHERE dept_id = 'D001'
    
    UNION ALL
    
    -- 递归：子部门
    SELECT d.dept_id, d.dept_name, d.parent_id, dt.level + 1
    FROM departments d
    INNER JOIN dept_tree dt ON d.parent_id = dt.dept_id
)
SELECT * FROM dept_tree ORDER BY level, dept_id;
⚠️ 递归CTE必须有终止条件（确保不会无限循环），大多数数据库默认递归深度限制为100或32767。
🔄

3️⃣ 多次复用同一子查询

如果同一个中间结果需要在主查询中多次引用（如JOIN、子查询条件），CTE可以避免重复代码。

-- 示例：统计高价值订单 & 关联商品明细，同时再次筛选同批客户
WITH high_value_orders AS (
    SELECT order_id, customer_id, total_amount
    FROM orders
    WHERE total_amount > (SELECT AVG(total_amount) FROM orders)
)
SELECT h.order_id, h.total_amount, p.product_name, oi.quantity
FROM high_value_orders h
JOIN order_items oi ON h.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
WHERE h.customer_id IN (SELECT customer_id FROM high_value_orders);
👓

4️⃣ 临时视图（一次性中间表）

某些中间结果只在当前查询有效，不值得创建持久化视图，CTE完美胜任。

WITH employee_stats AS (
    SELECT emp_id, emp_name, department,
           sales_amount,
           RANK() OVER (PARTITION BY department ORDER BY sales_amount DESC) as rank_in_dept,
           COUNT(*) OVER (PARTITION BY department) as dept_count
    FROM sales_records
    WHERE sale_date >= DATE '2026-01-01'
)
SELECT emp_name, department, sales_amount
FROM employee_stats
WHERE rank_in_dept <= dept_count * 0.1;   -- 各部门前10%精英

5️⃣ 数据去重 & 分组首条记录

-- 删除重复订单(保留最新)
WITH duplicates AS (
    SELECT order_id, order_date,
           ROW_NUMBER() OVER (PARTITION BY order_no ORDER BY order_date DESC) as rn
    FROM orders
)
DELETE FROM orders
WHERE (order_id, order_date) IN (
    SELECT order_id, order_date FROM duplicates WHERE rn > 1
);

-- 获取每个产品分类下销量最高的产品
WITH ranked AS (
    SELECT product_id, category, sales,
           ROW_NUMBER() OVER (PARTITION BY category ORDER BY sales DESC) as rn
    FROM products
)
SELECT * FROM ranked WHERE rn = 1;

6️⃣ 行列转换 / 数据透视

WITH sales_data AS (
    SELECT product_id, month, amount
    FROM monthly_sales
    WHERE year = 2026
)
SELECT product_id,
       MAX(CASE WHEN month = 1 THEN amount END) as Jan,
       MAX(CASE WHEN month = 2 THEN amount END) as Feb,
       MAX(CASE WHEN month = 3 THEN amount END) as Mar
FROM sales_data
GROUP BY product_id;

7️⃣ 分页查询（兼容模式）

在不支持OFFSET FETCH的旧版MySQL/SQL Server 2008等，利用ROW_NUMBER实现高效分页。

WITH numbered AS (
    SELECT t.*, ROW_NUMBER() OVER (ORDER BY create_time) as rn
    FROM large_table t
    WHERE status = 'active'
)
SELECT * FROM numbered
WHERE rn BETWEEN 11 AND 20;  -- 第2页，每页10条
📈

8️⃣ 金融分析 · 移动平均 & 累计值

WITH daily_sales AS (
    SELECT sale_date, SUM(amount) as total_amount
    FROM orders
    GROUP BY sale_date
)
SELECT sale_date, total_amount,
       AVG(total_amount) OVER (ORDER BY sale_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as ma_7days,
       SUM(total_amount) OVER (ORDER BY sale_date) as cumulative_revenue
FROM daily_sales
ORDER BY sale_date;

CTE vs 子查询 vs 临时表 对比

特性CTE子查询临时表
可读性⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐
单语句内复用✅ 多次引用❌ 不可重用✅ 可重用(但需单独创建)
递归能力✅ 支持递归❌❌
性能(大数)中等(优化器决定)差(多次执行)好(可建索引)
作用域单个SQL单个SQL会话/事务
适用数据量中小型/中型小型超大
💡 提示：CTE注重代码优雅，临时表注重性能（尤其多次引用+大量数据）。可以混合使用！
⚡

性能调优 & 注意事项

  • ✅ 优点：逻辑清晰、支持递归、一次定义多次使用，便于团队协作维护。
  • ⚠️ 潜在问题：部分数据库（老版本SQL Server、某些MySQL版本）CTE可能被多次物化，导致额外开销；
    递归CTE深度过大可能影响性能或超过递归限制。
  • 🚀 优化策略：
    • 如果CTE被多次引用且数据量巨大 → 考虑物化提示（Oracle: /*+ MATERIALIZE */）或改用手动临时表。
    • PostgreSQL 12+ 优化器会自动物化CTE，但可通过 OFFSET 0 技巧控制。
    • SQL Server: 如果CTE被多次JOIN，推荐用临时表代替。
    • 递归CTE务必为递归成员添加 WHERE 终止条件，防止无限循环。
-- Oracle 强制物化示例
WITH /*+ MATERIALIZE */ large_cte AS (SELECT ... )
SELECT * FROM large_cte;

-- SQL Server 用临时表替代高频CTE
SELECT * INTO #temp FROM ( ... ) t;
SELECT * FROM #temp t1 JOIN #temp t2 ...

真实场景案例：铁路机车检修节点任务

复杂业务逻辑：三表关联 + 条件判断 + 格式化输出，CTE让整体一目了然。

WITH node_task_data AS (
    SELECT DISTINCT t.KC_ID, t.MP_ID, n4.pk_id AS NODE_ID,
           nvl(d.gw, t.NODE_NAME) AS NODE_NAME,
           CASE WHEN d.GW IS NULL THEN t.PK_ID END AS TARGET_NODE_ID,
           d.KG_BZ, d.GW, t.DAY
    FROM OP_LOCO_JX_NODE t
    INNER JOIN OP_LOCO_JX_TASK d ON t.PK_ID = d.NODE_ID
    LEFT JOIN OP_LOCO_JX_NODE n4 ON n4.KC_ID = t.KC_ID AND n4.JOB_TYPE = 4
    WHERE t.JOB_TYPE IN (10, 750, 751) AND d.KG_BZ IS NOT NULL
)
SELECT KC_ID, MP_ID, NODE_ID,
       KG_BZ || '[' || NODE_NAME || ']' AS GNAME,
       '指定岗位工作人员' AS task_content,
       '00' AS status,
       TARGET_NODE_ID, KG_BZ, '0' AS is_gz, GW
FROM node_task_data
ORDER BY DAY, KG_BZ, GW;
📌 为什么用CTE？将繁杂的关联过滤封装，主查询只关注展示逻辑，后续扩展排序、聚合都非常干净。

什么时候选用 CTE？

✅ 推荐使用
• 子查询超过2层嵌套
• 同一子查询需重复使用
• 树形/递归查询（组织、菜单）
• 分步数据处理（ETL风格）
• 想要增强SQL可维护性
⚠️ 谨慎使用
• 千万级大表且复用多次
• 极简单层查询没必要
• 需要索引加速中间结果（改用临时表）
• 数据库版本太老可能有坑
💎 设计原则：优先保证代码清晰、可维护，遇到性能瓶颈再分析是否替换为临时表或物化方式。绝大多数业务查询，CTE足够优雅高效。
